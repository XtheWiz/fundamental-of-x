Lesson 07

Password Hashing

Lesson 1 said hashing is one-way. Surely that means SHA-256 is fine for storing passwords? No. Hashing speed is a curse here: an attacker with your hash database can try billions of guesses per second. Password hashing is about making hashing deliberately slow.

The problem with fast hashes

If your database leaks (it happens), the attacker has every hash. They can't reverse them, but they can guess — try common passwords, hash them, look for matches. Modern GPUs do tens of billions of SHA-256 hashes per second. Most passwords are weaker than that throughput.

So:

  • Slow hashes: bcrypt, scrypt, Argon2. Designed to be slow — a single hash takes ~100ms. That's ~10 guesses/sec instead of billions.
  • Salts: a random value per user, mixed in before hashing. Means the attacker can't pre-compute a rainbow table — they have to brute-force every user separately.
  • Memory-hard (scrypt, Argon2): also require lots of RAM, which makes GPU/ASIC attacks much harder.

Try it

Type a password. See what happens if it's stored as SHA-256 vs bcrypt, when an attacker tries to crack it.

Key takeaways

  • Never store passwords directly, and never with a fast general-purpose hash like SHA-256.
  • bcrypt, scrypt, Argon2 are the right choices. Argon2id is the modern default.
  • Always use a per-user salt — every password hashing library does this for you.
  • The "work factor" / "cost parameter" should be tuned so each login takes ~100-300ms on your hardware. Bump it up over time as hardware gets faster.
