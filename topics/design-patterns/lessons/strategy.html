Lesson 02

Strategy

"Same operation, swappable algorithm." Sorting orders, payment methods, compression codecs, pricing tiers. The Strategy pattern says: pull the variable bit out into an object (or function) and pass it in.

The shape

  • A context needs to do some operation (sort, compress, charge).
  • The operation has multiple variants. Instead of an if/else ladder, the context holds a reference to a strategy object that implements the operation.
  • Callers (or some config layer) pick which strategy to plug in.

In languages with first-class functions, the "strategy object" is usually just a function. sort(items, comparator) is Strategy in disguise — and every language has it built in.

Try it

Without Strategy: an if/else ladder. With Strategy: a function/object passed in. Same data, three orderings.

Key takeaways

  • The clearest sign you need Strategy: an if (type === …) ladder that keeps growing.
  • In languages with closures/lambdas, Strategy is one line: pass the function. No need for a full class hierarchy.
  • Don't use Strategy if there's only one current variant. "What if we add another?" is a future tax — pay it when you need it.
  • Closely related: State (strategy that can change at runtime), Template Method (skeleton with strategy holes), Visitor (strategy that walks a structure).
