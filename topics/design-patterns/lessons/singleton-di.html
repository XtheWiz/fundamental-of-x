Singleton & Dependency Injection

Singleton is the GoF pattern most likely to start an argument. It looks tidy — one global instance — but it's testability poison. Dependency Injection is the modern alternative, and it's not really a "pattern" so much as a habit.

The Singleton temptation

"There's only one logger / config / cache — surely it should be a Singleton?" The argument feels right. The cost shows up in tests, where every test starts up the same global instance, shares state with every other test, and can't be substituted for a fake.

Dependency Injection

Instead of finding your dependencies (singleton lookup, global), have them passed in. The class takes its collaborators as constructor arguments (or function arguments). Production wires the real ones; tests pass fakes.

DI doesn't require a framework. "Just pass the thing as an argument" is enough for most code. Frameworks (Spring, NestJS, Dagger) automate the wiring at scale — useful but optional.

Key takeaways

  • Singleton ≈ global variable with extra steps. Same problems: shared mutable state, test fragility, hidden dependencies.
  • DI says: explicit dependencies are testable dependencies. The function signature tells you what the function needs.
  • "There's only one of these" is a runtime decision (you instantiate it once), not a language enforcement. The composition root sets up your one instance.
  • Exceptions: stateless utilities (Math, Crypto), language-level singletons (modules), and certain framework constructs. For most app code, prefer DI.
