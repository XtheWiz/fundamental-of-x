Factory & Builder

Two patterns for the question "how do I create this object?" Factory hides which class to instantiate; Builder composes a complicated thing step by step. They solve different problems but get confused often.

Factory — choose the class for me

Use when the right concrete type depends on input you don't want callers to switch on. Loading an image? ImageFactory.from(path) returns a PNG, JPEG, or WebP object — the caller doesn't care.

Builder — compose me step by step

Use when an object has many optional parameters or a multi-step construction process. Builders give you a fluent API: new HttpRequestBuilder().url(…).header(…).body(…).build(). More readable than a constructor with 12 arguments — and forces validation at build().

Key takeaways

  • Factory hides which. Builder hides how.
  • If your "factory" only ever returns one type, you don't need it. Just call new.
  • Builders shine for immutable objects with many optional fields. Less great in languages with named/default arguments (Python, Kotlin) — those make builders mostly unnecessary.
  • "Abstract Factory" is the GoF distinction — a factory of factories. Genuinely useful but rare; usually overkill.
