Repository & Unit of Work

The two patterns that organize data access in most enterprise codebases. Repository abstracts where the data lives; Unit of Work batches changes into one atomic commit.

Repository — collection-like access to a data source

UserRepository.byId(42), OrderRepository.where(status: 'open'). Looks like a collection in memory; underneath it could be SQL, REST, an in-memory map (for tests), or all three.

The point is: business logic doesn't write SQL. It talks to repositories. Swapping persistence layers becomes a contained change.

Unit of Work — one logical transaction

Within one business operation you might create three objects and update two more. A Unit of Work tracks all changes in memory, then commits them in a single DB transaction.

Most ORMs (Hibernate, SQLAlchemy, Entity Framework) have a built-in UoW called the session or context. session.commit() writes everything; session.rollback() throws it away.

Key takeaways

  • Repository lets you test business logic with a fake repo. No DB needed for unit tests.
  • UoW gives you transactional semantics in code: do many things, commit atomically.
  • Anti-pattern: repository-per-table. Repositories should match aggregates, not schemas. OrderRepository handles orders and their line items.
  • For small projects with simple persistence, these patterns are overkill. They earn their keep when the data model is rich and the persistence layer might change.
