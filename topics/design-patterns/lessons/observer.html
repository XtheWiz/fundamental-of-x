Observer & Pub-Sub

"When X happens, notify everyone who cares." Without knowing who they are. The Observer pattern (and its big-cousin Pub-Sub) powers every event-driven system you've ever used.

The shape

  • A subject (also called publisher) maintains a list of observers (subscribers).
  • Subscribers register via subject.subscribe(fn) — they get a callback.
  • When something happens, the subject calls each observer's callback with the event.
  • The subject doesn't know the observers' identities. Loose coupling.

Where you'll see it

  • DOM events: element.addEventListener('click', …)
  • Node.js EventEmitter, browser CustomEvent, RxJS observables
  • UI frameworks: React's useState + re-render is observer-like; MobX/Svelte stores are direct
  • Message queues & event buses: Kafka, RabbitMQ, Redis pub/sub (distributed Observer)
  • Postgres LISTEN/NOTIFY, mongoDB change streams

Try it

A subject ("message-received") with three observers (badge, dropdown, page title). Publish events, see each react independently.

Key takeaways

  • Observer decouples sender from receivers. Add a new reactor without touching the publisher.
  • Failure mode: observers run synchronously. A slow or throwing observer can block / break the publisher. Many real systems queue events and dispatch async.
  • Pub-Sub is Observer scaled: an event bus mediates between many publishers and many subscribers, often across processes/machines (Kafka).
  • Beware "spaghetti subscribers" — large apps with hundreds of unrelated reactors are hard to debug. Centralize event types (an event catalog) helps.
