Modern Architecture Patterns

Patterns the original GoF book never saw coming. Most of these organize whole applications or service interactions rather than single-class structure — but the GoF-style "named solution to a recurring problem" lens still applies.

What changed

GoF (1994) targeted single-process OO code. The last three decades added: networks everywhere, microservices, async event-driven systems, horizontal scale, distributed failure modes, and the realization that most apps are CRUD over a database. New patterns address those realities.

The lineup

  • MVC / MVVM — separation of presentation, state, and logic. Half the UI frameworks ever built use a variant.
  • CQRS (Command Query Responsibility Segregation) — split the write model from the read model. Different shapes for different uses.
  • Event Sourcing — don't store state; store the history of changes that produced it. Replay to reconstruct.
  • Circuit Breaker — when an upstream is failing, stop hammering it. Fail fast for a window, then probe.
  • Saga — distributed transaction as a sequence of local steps with compensations on failure. The modern replacement for 2PC.

Try it

Pick a pattern. See its shape, a use case, and a sample sequence.

Key takeaways

  • "Modern" patterns mostly address distributed systems. They presume a network between components, and they have to deal with partial failures.
  • Each is more expensive than its alternative. CQRS doubles your data model; Event Sourcing complicates queries; Saga code has compensation paths everywhere.
  • Don't reach for them without need. "We might need this someday" is the highest tax in software — write the simple thing first, refactor when the data demands it.
  • Many of these come with named books (Vaughn Vernon's, Greg Young's, Pat Helland's). The patterns are deeper than this overview — start here, dig there.
