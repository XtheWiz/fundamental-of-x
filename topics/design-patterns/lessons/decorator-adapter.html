Decorator & Adapter

Both wrap existing objects. Decorator adds behavior while preserving the interface; Adapter changes the interface without altering behavior. Confused often, actually doing different jobs.

Decorator — same shape, extra behavior

Wrap an object in another object that implements the same interface, but adds logging, caching, retries, encryption, etc. before/after delegating to the wrapped one. Stack as many as you like. The classic example is HTTP middleware: compress(log(auth(handler))) — each layer wraps the next.

Adapter — different shape, same behavior

You have a thing with one API; callers want a different API. Adapter is the translator. Common when wrapping a third-party library or upgrading providers.

Key takeaways

  • Decorator preserves the interface — you can keep stacking. Adapter changes the interface — usually one layer.
  • Express middleware, Python decorators (the syntax), Rust's tower service layers — all are Decorator pattern with language sugar.
  • Adapters often hide ugly: legacy libraries, vendor SDKs, old code you can't change. They keep the rest of your code clean.
  • Don't over-decorate. Five layers of wrappers is hard to debug. Same with adapters wrapping adapters — that's an architecture smell.
