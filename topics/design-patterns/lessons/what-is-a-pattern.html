What is a Pattern?

"Design Pattern" sounds important. Let's take the religion out of it: a pattern is a name for a solution to a recurring problem. That's it. Shared vocabulary, not gospel.

The 1994 origin story

"Design Patterns: Elements of Reusable Object-Oriented Software" (Gamma, Helm, Johnson, Vlissides — "the Gang of Four") catalogued 23 patterns in 1994. The book gave us shared vocabulary for things experienced devs were already doing, so they could discuss design without having to re-explain every time.

That was the win. The unintended consequence:

"Pattern hammering." Junior devs treating the catalog as a checklist. Code stuffed with Factories, Visitors, and Singletons solving problems the code didn't have. The point of patterns is to talk about solutions you already needed — not to introduce solutions to problems you don't.

Patterns that became language features

Many GoF patterns were workarounds for limitations of 1994's mainstream languages (C++, Java). Modern languages often have built-in equivalents:

  • Iterator → for…of, generators, ranges, streams
  • Strategy → first-class functions / lambdas
  • Command → functions-as-values, closures
  • Observer → events, signals, reactive streams
  • Visitor → pattern matching, sum types
  • Singleton → module-level state, dependency injection

That doesn't make the patterns obsolete — knowing them helps you understand legacy code and reach for the named solution when languages don't provide it natively.

Key takeaways

  • A pattern is a name. The value is shared vocabulary, not the mechanics.
  • "What pattern should I use?" is usually the wrong question. The right one is "what problem am I trying to solve?"
  • Modern languages erode some patterns. Knowing what got absorbed (Strategy → lambdas) sharpens your design instincts.
  • Spend equal energy on anti-patterns — God objects, deep inheritance, anemic models. Knowing what to avoid is half the skill.
