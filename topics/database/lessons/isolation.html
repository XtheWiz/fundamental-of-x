Transaction Isolation

When transactions run at the same time, weird things can happen. Isolation levels are dials that decide which weird things you tolerate — and how much performance you pay to avoid them.

The three phenomena

  • Dirty Read: T2 reads a value T1 wrote — but T1 then rolls back. T2 saw a value that never officially existed.
  • Non-Repeatable Read: T2 reads row R. T1 commits an update to R. T2 reads R again — different value.
  • Phantom Read: T2 queries WHERE x > 5. T1 commits a new row matching the predicate. T2 re-runs — a new row appeared.

The four levels

LevelDirtyNon-Rep.Phantom
Read Uncommitted❌❌❌
Read Committed✅❌❌
Repeatable Read✅✅❌*
Serializable✅✅✅

* The SQL standard allows phantoms at Repeatable Read. PostgreSQL's "Repeatable Read" actually prevents them too via snapshot isolation. Implementations vary.

Try it

Pick a phenomenon and an isolation level. Watch the two transactions interleave — see what the database lets through.

Key takeaways

  • Higher isolation = more correctness, less concurrency. Pick the lowest level that's safe for your workload.
  • Most databases default to Read Committed (Postgres, Oracle) or Repeatable Read (MySQL).
  • Real databases implement isolation with locks, snapshots (MVCC), or both — not by literally serializing.
  • The named levels come from the SQL standard, but implementations diverge. Read your DB's docs.
