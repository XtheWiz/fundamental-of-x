Query Execution Pipeline

SELECT name FROM users WHERE city = 'Tokyo'; looks simple. But between you typing that and getting rows back, the database does a lot. Here's the assembly line.

Five stages

  1. Parse — read the SQL text, check syntax, build an AST.
  2. Bind / Analyze — resolve names against the catalog. users exists? city is a real column?
  3. Optimize — pick a plan. Use an index? Which join order? This is where databases earn their reputation.
  4. Execute — run the chosen physical plan: scan, seek, filter, sort, join, aggregate.
  5. Return — stream rows back to the client.

Try it

Pick a query and step through how a database would process it.

Key takeaways

  • The optimizer is the magic. Two semantically identical queries can have wildly different plans — and costs.
  • Indexes don't speed up queries; they give the optimizer options. The optimizer picks them when statistics say they help.
  • EXPLAIN in any real database shows you the chosen plan. Read it.
  • Query rewriting (constant folding, predicate pushdown, join reordering) happens before execution.
