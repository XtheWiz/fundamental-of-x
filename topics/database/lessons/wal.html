Write-Ahead Log (WAL)

How does a database survive a sudden power cut? The trick is disarmingly simple: write to the log before you touch the data. The log can be replayed; corrupted data files cannot.

Why we need it

Writing every change directly to the data file is too slow — disk seeks are expensive, and data pages might be far apart. Instead, databases keep the "real" data in memory and stream a stripped-down record of every change to a sequential log file.

The contract:

  • Before any modification is "committed", its log record is on disk.
  • The actual data page can be flushed to disk much later (or never, if it's already correct on next startup).
  • On crash, the database scans the log and replays committed changes; discards uncommitted ones.

Try it

Step through a transaction one operation at a time. Press Crash at any point and watch recovery.

Key takeaways

  • WAL = sequential append-only log of every change, fsync'd before commit returns.
  • Data pages are lazy — written in batches long after the commit. Memory holds the "real" view.
  • Recovery has two phases: redo committed entries; undo uncommitted ones.
  • Same idea, different names: PostgreSQL pg_wal, MySQL InnoDB redo log, SQLite journal, RocksDB WAL.
