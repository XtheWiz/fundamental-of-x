ACID Properties

ACID is a promise a database makes about transactions. Four letters, four guarantees. Each one protects against a specific kind of disaster — try breaking them yourself.

The four letters

A transaction is a group of operations that should behave like a single one. ACID is what makes that "behave like one" claim believable:

  • A — Atomicity: All operations succeed, or none of them do. No half-done transactions.
  • C — Consistency: The database moves from one valid state to another. Constraints are never violated.
  • I — Isolation: Concurrent transactions don't trip over each other (full topic in Lesson 4).
  • D — Durability: Once committed, a write survives crashes (full topic in Lesson 5).

Try it

Pick a letter and run the demo. Try to break the guarantee — watch the database catch you.

Key takeaways

  • ACID is about transactions, not the whole database. A non-transactional write isn't protected.
  • Atomicity = "no half-done" — rollback brings you back to the pre-transaction state.
  • Consistency = "constraints held" — uniqueness, foreign keys, CHECKs are enforced.
  • Isolation and Durability are big enough topics to get their own lessons.
