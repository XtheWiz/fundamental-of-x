Row vs Column Storage

The same table can be laid out on disk in two very different ways. That choice decides whether you're fast at "give me this whole row" or "give me the sum of this one column across millions of rows."

The intuition

Disk reads happen in pages (4–8KB chunks). The database can't read just a single cell — it has to load the whole page containing that cell. So the layout of data on those pages decides how much useless data you drag along with the useful bits.

  • Row store: each page packs whole rows together. Great for "give me row #42, all columns."
  • Column store: each page packs one column's values together. Great for "sum the price column across 10M rows."

Try it

Pick a query. The cells the database has to read get highlighted; the ones it skips are hatched out.

Key takeaways

  • Row stores (Postgres, MySQL InnoDB) win for OLTP — short queries that touch many columns of a few rows.
  • Column stores (ClickHouse, Redshift, Parquet) win for OLAP — aggregations that touch one column of many rows.
  • The same data; only the on-disk arrangement differs. There is no "better" — only "better for which workload."
  • Column stores also compress brilliantly: similar values sit next to each other.
