Sharding & Partitioning

One machine can only hold so much data and do so many queries per second. Sharding is the trick of splitting the data across many machines. The interesting question: how do you decide which row goes where?

The setup

You have 60 user rows and 4 shard nodes. Pick a strategy:

  • Hash: shard = hash(user_id) % N. Even distribution, but range queries hit every shard.
  • Range: ids 1–15 → shard A, 16–30 → B, etc. Range scans are cheap but hot ranges = hot shards.
  • Geo: by region. Locality wins, but populations are uneven (Tokyo dominates).

Try it

Key takeaways

  • The choice of shard key is the most important decision in a sharded system. Once data is placed, moving it is painful.
  • Hash = balanced load, but loses locality (range queries fan out).
  • Range = great for range scans, but creates hot shards.
  • Geo = low latency for users in the same region, but skewed populations create hot shards too.
  • "Partitioning" in Postgres/MySQL is the same idea but within one database; "sharding" usually means across machines.
