Anatomy of HTTP

HTTP is a text protocol that humans can read. Three parts to a request, three to a response, and a vocabulary of verbs and status codes that says everything about what you want and what the server did about it.

Request and response, side by side

Every HTTP request has three parts:

  • Request line: VERB /path HTTP/1.1 — the intent.
  • Headers: Key: Value pairs — metadata (auth, content type, caching hints).
  • Body: the payload (optional, depends on the verb).

Every response has three parts too:

  • Status line: HTTP/1.1 200 OK — what happened, in three digits.
  • Headers: metadata going the other way.
  • Body: the response payload (HTML, JSON, binary, whatever).

The vocabulary

Verbs say what you want done. Status codes group by leading digit: 2xx success, 3xx redirect, 4xx "you screwed up," 5xx "I screwed up." Most APIs use perhaps a dozen of each in practice.

Try it

Build a request. See it printed in HTTP's wire format. See what response makes sense for the given verb.

Key takeaways

  • Verbs encode intent. GET should never modify data; DELETE should never return data. Mismatched verbs confuse caches, retries, and humans.
  • Status codes are an API. Returning 200 for a failed login is a bug — clients can't tell apart success from failure without parsing the body.
  • Headers carry the contract. Content-Type, Authorization, Accept, Cache-Control — most of HTTP's power is in headers.
  • HTTP/1.1 is text; HTTP/2 and /3 are binary. The semantics are identical — the difference is on the wire (covered in the Internet topic).
