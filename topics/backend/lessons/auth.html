Sessions & Auth

How does your server know it's still you on the second request? HTTP is stateless — every request stands alone — so we have to carry identity along. Cookies, tokens, JWTs, and OAuth are all answers to "how do we do that safely?"

The three answers

  • Sessions + cookies: server keeps state, hands client a random session ID via cookie. Simple, scales fine if you have a shared session store. Default in most web frameworks.
  • Tokens (JWTs): server signs a token containing the user info. Client carries it. No server-side state to look up — verify the signature and trust the contents. Easy to scale, hard to revoke.
  • OAuth: not really auth — it's delegated authorization. "Let this app act on my behalf at Provider X." Used everywhere as a login mechanism via "Sign in with…"

The classic Authorization Code flow has three parties — your app, the user's browser, and an identity provider like GitHub. Step through each message.

Key takeaways

  • OAuth's safety comes from never letting your app see the user's password. Only the provider does.
  • The code from the redirect is one-time, short-lived, and only useful when combined with your secret client_secret.
  • The state parameter is anti-CSRF: your app sets it, checks it came back unchanged. Skipping it is a known vulnerability class.
  • "Sign in with X" buttons are OAuth under the hood — the access token gives your app permission to call /me on the provider, and from that you create a local session.
