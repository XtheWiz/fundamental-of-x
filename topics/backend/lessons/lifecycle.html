Lesson 08

Request Lifecycle

We've covered the parts. Now watch them in concert: a browser making a request travels through CDN, load balancer, reverse proxy, app server, cache, and database — six possible stops and many possible shortcuts.

The full journey

A typical modern stack, in order of contact:

  1. Browser — checks its own cache first. May skip everything else.
  2. CDN — serves static/cacheable responses from the edge node nearest you.
  3. Load Balancer — picks a healthy backend instance.
  4. Reverse Proxy — terminates TLS, routes by path, may also cache.
  5. Application Server — runs your code: middleware, auth, business logic.
  6. App Cache (Redis/Memcached) — pre-computed answers for hot queries.
  7. Database — the source of truth, only touched when nothing else has the answer.

Every layer is a chance for a hit (fast) or a miss (continue to next layer). Optimize for hit rates at the outermost layers — that's where the biggest wins are.

Try it

Send a request. Toggle which layers hit their cache. See where the milliseconds go.

Key takeaways

  • The fastest request is the one served from the layer closest to the user. A CDN hit at the edge skips your origin entirely — sub-50ms responses globally.
  • The slowest request hits every layer down to the database. That's still usually <200ms on a healthy system.
  • Most production "performance work" is about moving the hit rate up the stack: more CDN hits, more proxy hits, more app cache hits.
  • Each layer can fail independently. Observability (per-layer metrics + traces) is how you locate problems quickly.
