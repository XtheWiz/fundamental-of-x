Caching

The fastest request is the one you don't make. A modern web stack has cached copies of the same data in at least four places — and the trick to making it work is knowing what to cache, where, for how long, and how to invalidate.

The four layers

  1. Browser cache — per-user, controlled by Cache-Control headers. Free, but only helps repeat visitors.
  2. CDN — shared across all users in a region. Cuts trips to origin entirely for static content.
  3. Reverse proxy (nginx, Varnish) — at the edge of your infrastructure. Can cache even dynamic responses with the right hints.
  4. Application cache (Redis, Memcached) — inside the app, in front of the database. Pre-computed views, hot keys, computed results.

And implicitly: database query cache below all of these.

Try it

Send a request and watch it cascade through each layer. Each cache has its own TTL. Try invalidating specific layers and observing what happens.

Key takeaways

  • Caches save latency, bandwidth, and load. A 99% CDN hit rate means your origin sees 1% of the traffic — that's an order-of-magnitude saving.
  • Cache invalidation is the hard part. "There are only two hard things…" — Phil Karlton. Most cache bugs are stale data, not misses.
  • TTLs are a tradeoff: long = fewer origin hits, more stale data. Short = more origin hits, fresher data. Tune per content type.
  • HTTP gives you the tools: Cache-Control: public, max-age=3600, ETag, Last-Modified, Vary. Use them deliberately.
