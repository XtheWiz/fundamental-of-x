Reverse Proxies

nginx, HAProxy, Envoy, Cloudflare. The same job: stand in front of your real servers and pretend to be them. Done well, the outside world never knows your backend exists.

What a reverse proxy actually does

  • TLS termination. The cert lives on the proxy. Backends speak plaintext HTTP on the private network. Saves backend CPU and simplifies cert rotation.
  • Path routing. /api/* → api-server, /static/* → static-server, fallthrough → web-server. One public hostname, many backends.
  • Header rewriting. Adds X-Forwarded-For, X-Real-IP, X-Forwarded-Proto so the backend can still see the original client info.
  • Load balancing. Across multiple instances of the same backend (full lesson next).
  • Security buffer. Hides backends from direct internet exposure. Often combined with WAF rules.
  • Compression, caching, rate limiting — all opt-in extras.

Try it

Send a request to the proxy. See the routing rule that matched, the header it added, and the backend it forwarded to.

Key takeaways

  • Reverse proxy = "I'm pretending to be the backend." Forward proxy = "I'm pretending to be the client." Same machinery, opposite direction.
  • Almost every production website is behind one — even if you don't explicitly run nginx, your CDN is one.
  • Without X-Forwarded-For, your backend logs would show all requests coming from the proxy's IP. Always trust this header carefully (it's user-controllable upstream).
  • TLS termination is the most common reason to introduce one. Even single-backend sites benefit.
