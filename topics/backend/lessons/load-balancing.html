Load Balancing

One way to handle 10× traffic: run 10× the servers and split the work. That "splitting" is what a load balancer does. The magic is in how it splits — and not all strategies are equal under uneven load.

The four strategies

  • Round-robin — backend 1, 2, 3, 4, 1, 2, … Simple, fair if all backends are equal. Falls apart when one is slow.
  • Least connections — send the next request to the backend with the fewest active connections. Self-corrects when one backend is slow.
  • IP hash — hash the client's IP, mod by N backends. Same client always lands on the same backend (sticky sessions without explicit cookies).
  • Random — pick uniformly at random. Surprisingly close to round-robin at scale; simpler in distributed LB setups.

Try it

Send 20 requests. Pick a strategy. Backend 3 is artificially slow — see how each strategy reacts.

Key takeaways

  • Round-robin is fine until it isn't — once backends drift apart in capacity, you get pile-ups.
  • Least-connections is the right default for most web workloads. It naturally avoids slow backends.
  • IP hash is necessary when you need session affinity without explicit session storage. Use sparingly — failed backends mean some users have a bad day.
  • Health checks are inseparable from LB. A strategy is only as good as its decision to skip dead backends.
