REST & API Design

REST is a style, not a standard. The good news: most of it is just "use HTTP the way it was meant to be used." The bad news: a lot of APIs in the wild don't.

The principles

  1. Resources, not verbs in URLs. /users/42, not /getUser?id=42. The HTTP verb is the verb.
  2. Use methods correctly. GET reads. POST creates. PUT replaces. PATCH updates. DELETE deletes.
  3. Stateless. Every request carries everything the server needs. No session affinity (except via cookies/tokens carried explicitly).
  4. Predictable status codes. 2xx success, 3xx redirect, 4xx client problem, 5xx server problem.
  5. Idempotency where promised. A retry of PUT/DELETE shouldn't double-delete. POST often isn't idempotent — design around it (idempotency keys).

Try it

Score yourself on six API-design choices. Each scenario gives two options. Pick the more RESTful one.

Key takeaways

  • Most "wrong" APIs aren't wrong-wrong — they work — but they're harder to cache, retry safely, and reason about.
  • "REST" is fuzzy. RESTful means following the spirit (resources + HTTP verbs); HATEOAS is the strict academic version that almost nobody actually does.
  • For internal/private APIs, gRPC or GraphQL may fit better than REST. Pick the tool for the job — REST isn't a religion.
  • Versioning is unavoidable. Prefix with /v1/, use Accept-Version headers, or both. Decide before launch — it's hard to change later.
