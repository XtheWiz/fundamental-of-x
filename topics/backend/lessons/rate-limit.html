Rate Limiting

Every public API has limits. They protect you from runaway clients, accidental DOS, and the occasional malicious user. The most common implementation is the token bucket — and it's beautifully simple.

The token bucket

Imagine a bucket. It holds up to capacity tokens. Tokens drip in at refill rate per second. Every incoming request takes one token. If the bucket is empty, the request is denied — usually with HTTP 429 "Too Many Requests."

This gives you two things at once:

  • Sustained rate: long-run throughput is bounded by the refill rate.
  • Burst tolerance: a client that's been quiet can briefly burst up to capacity requests in a row.

Try it

Adjust the bucket's capacity and refill rate. Click "Request" repeatedly. Try a burst. Watch the bucket drain and refill.

Key takeaways

  • Rate limit per user/IP/key, not globally. Otherwise one heavy user starves the rest.
  • Always include a Retry-After header on 429s — clients can use it to back off intelligently.
  • Leaky bucket is the other classic algorithm — same intuition with smoother output. Good for shaping outbound traffic.
  • Sliding window counters (e.g. Redis-based) are popular for distributed services because they're simple to implement across many LB instances.
