Backpressure & Flow Control

Producer at 10,000 messages/sec. Consumer at 1,000/sec. Where do the extra 9,000 go each second? Buffer (until you run out of disk), drop (lose data), slow producer down (apply backpressure), or fail loudly. Picking which is a design decision, not a default.

The four responses

  • Buffer: store the overflow on disk. Works until you run out. Adds latency proportional to queue depth.
  • Drop: explicit policy ("oldest first," "lowest priority," "tail-drop"). Lossy, fast, predictable. Used for telemetry, metrics.
  • Backpressure: signal the producer to slow down. The healthy response. Requires producer cooperation.
  • Fail: producer's writes start returning errors. Worst UX, simplest implementation.

How brokers actually do it

  • Kafka: relies on disk + retention. Producers don't get backpressured unless the broker hits storage limits. Consumers lag silently.
  • RabbitMQ: built-in flow control. When memory hits a watermark, producers are throttled at the connection level.
  • SQS: has hard message-count limits; producers get errors past them.
  • NATS: drop-by-default for JetStream; pure pub-sub drops if no subscribers.
  • TCP itself implements backpressure via window size — when the receive buffer fills, the sender stops.

Try it

Run a sustained mismatch between producer and consumer rates under each response.

Key takeaways

  • Consumer lag is the most important metric. If it grows unbounded, you have a backpressure problem.
  • Choose the response based on data value. Metrics can drop. Payments can't.
  • "Auto-scaling consumers" is the most common backpressure response in cloud systems — add more workers when lag grows.
  • Reactive Streams (Project Reactor, RxJS) bring backpressure to in-process pipelines too. Same idea, different scale.
