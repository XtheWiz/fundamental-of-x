Queues vs Streams

"Messaging" hides two different shapes underneath. A work queue (RabbitMQ, SQS) is for distributing tasks. A stream (Kafka, Kinesis) is an append-only log replayable by many readers. Same word, very different mechanics.

Work queue: "do this task"

  • Each message is a unit of work to be done by one worker.
  • Workers compete for messages; once consumed, it's gone.
  • Used for: email sending, image resizing, payment processing — anything where exactly one worker should do the job.
  • Failure: message returns to the queue (or to a DLQ — lesson 7).

Stream: "here's what happened, in order"

  • Messages are events on a log. Many independent readers can consume the same log.
  • Reading doesn't delete; consumers track their own offset (which position they've reached).
  • Used for: event sourcing, analytics, change data capture, stream processing.
  • You can rewind. Add a new consumer six months later — it starts from the beginning.

The blurred middle

Modern brokers don't strictly fit one mold:

  • RabbitMQ Streams: log-style on top of queue infrastructure.
  • Kafka Queues (KIP-932): work-queue-style competing consumers on top of a log.
  • SQS: pure work queue, no replay.
  • Redis Streams: hybrid; readers consume by group (work-queue) or independently (log).

The mental model still helps: are you distributing tasks or recording events?

Try it

Run the same 10 messages through both shapes. See how consumption differs.

Key takeaways

  • If only one consumer should ever process a message: queue. If many independent readers care: stream.
  • Streams give you replay for free — invaluable for analytics and debugging. Queues don't.
  • Streams use more disk (retention) but cheaper per message. Queues drop messages as soon as they're acked.
  • Don't pick a tool because it's trendy. Pick the shape that matches the data flow you actually have.
