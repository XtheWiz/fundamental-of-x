Ordering & Idempotency

Two questions every messaging system has to answer: in what order do messages arrive, and what happens if the same one shows up twice? The answers determine how your consumer code has to be written.

Ordering — only within a partition

Kafka guarantees order within a partition, not across partitions. If you partition by user_id, all of a user's events stay in order — that's usually what you want.

If you partition by something else (round-robin, hash of event_id), related events for the same entity might arrive out of order. The "user updated email then requested password reset" sequence can flip. Pick the partition key carefully.

Idempotency — the antidote to duplicates

At-least-once delivery + retries = your consumer will see duplicates eventually. The fix isn't in the broker — it's in your code:

  • Idempotency key: every message has a unique ID. Consumer remembers IDs it's already processed (in a DB or cache), and skips repeats.
  • Idempotent operations: "set balance = 500" is naturally idempotent. "balance += 100" is not. Where possible, model state changes as absolute values.
  • Database constraints: a unique key on (user_id, order_id) prevents the second insert from succeeding.

Key takeaways

  • Partition by the entity whose order matters. user_id for user events; order_id for fulfillment steps.
  • Always assume duplicates can happen. Idempotent consumers are forgiving; non-idempotent ones eventually corrupt data.
  • Idempotency keys + a "processed message IDs" set is the simplest pattern. Bounded retention (e.g., 24h) prevents the set from growing forever.
  • Order-sensitive workflows (workflow engines, sagas) often need an explicit ordering layer above the broker — Kafka alone gives partition-local order, not global.
