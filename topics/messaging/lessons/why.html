Lesson 01

Why Messaging?

Service A calls Service B directly. Simple, fast, works great — until B is slow, or down, or being deployed. Putting a broker between them sounds like adding complexity, but it's actually subtracting coupling.

The coupling problem

Direct sync calls couple two services on every axis:

  • Temporal coupling: both must be up at the same time.
  • Capacity coupling: A's request rate must fit within B's throughput.
  • Failure coupling: B's outage cascades to A.
  • Schema coupling: A must speak exactly the API B exposes today.

For requests that need an answer right now (auth, payment), this is fine. For everything else — sending an email, processing an upload, indexing a document — it's overkill.

What a broker buys you

  • Decoupling in time. A writes a message; B reads it whenever. B can be slow, restarting, or temporarily gone.
  • Decoupling in capacity. A queue absorbs bursts. B drains at its own pace.
  • Decoupling in failure. If B crashes, messages wait. A doesn't know or care.
  • Fan-out. One message, many consumers. Add a new consumer without touching A.

The price: more moving parts (the broker), eventual consistency, no immediate "did it work?" feedback.

Try it

Simulate a producer (orders/sec) and a consumer (slower). Compare direct calls vs. a broker between them.

Key takeaways

  • Use sync calls for things that need an answer. Use messaging for things that can finish later.
  • Messaging trades immediate feedback for resilience. Users can't tell their email is queued vs sent — that's the point.
  • "We added Kafka and now everything's eventually consistent" is a common pain. Migrate carefully; some flows really do need sync.
  • The broker itself becomes a dependency. Pick one that fits your scale and ops bandwidth — covered in lesson 8.
