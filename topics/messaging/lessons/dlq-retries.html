Dead Letter Queues & Retries

A consumer crashes on the same message every time. Retry forever and the queue stops draining. Skip silently and lose data. The grown-up answer is exponential backoff plus a dead letter queue — quarantine the poison message and keep the line moving.

Why retry?

Most failures are transient: a network blip, a DB momentarily unavailable, a downstream service in the middle of a deploy. Retrying after a short wait fixes them transparently.

But some failures are permanent: a malformed message, a bug in the consumer code, a referenced record that no longer exists. No amount of retrying will help — and worse, an unbounded retry blocks every message behind the poison one.

The pattern

  1. On failure, increment a retry count and wait exponentially longer each time (1s → 2s → 4s → …).
  2. Add a jitter (random ±25%) to avoid retry stampedes when many consumers fail at once.
  3. After N retries, give up. Send the message to a dead letter queue (DLQ) for human review.
  4. Alert on DLQ growth — it's almost always a sign of a bug or upstream issue.

Try it

Process a stream where one message is poisonous. Configure retry count + backoff and watch outcomes diverge.

Key takeaways

  • Always cap retries. Unbounded retry is how systems wedge themselves.
  • Exponential backoff + jitter is the standard. Linear backoff causes thundering herds; constant backoff is too aggressive.
  • The DLQ is a triage tool, not a graveyard. If it grows, something needs investigating — alert on it.
  • Some brokers (SQS, RabbitMQ) have DLQs built in. Kafka requires you to write to a separate topic — same idea, more wiring.
