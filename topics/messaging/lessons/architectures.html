Architectures in Practice

Same primitives — topics, partitions, delivery semantics, DLQs — composed five very different ways. Pick the system that fits your data flow, not the one with the cool logo.

The lineup

All of these are "messaging." They differ in shape (queue vs stream), scale, semantics, and operational cost. The widget below lays them side-by-side along the dimensions covered in earlier lessons.

Key takeaways

  • Kafka is the heavyweight: high throughput, replayable log, complex to operate. Default for analytics + event streaming at scale.
  • RabbitMQ shines for work-queue patterns + complex routing (exchanges, topic routing). Lower throughput than Kafka, much easier ops.
  • SQS is the boring-and-reliable cloud option. No replay, no servers to run, fixed retention. Great for task queues if you're on AWS.
  • NATS / Redis Streams are lightweight alternatives. NATS wins for low-latency request/reply + edge use cases; Redis Streams for "I already run Redis."
  • Don't try to make Kafka do what RabbitMQ does, or vice versa. Use the right tool for the shape of your traffic.
