Topics, Partitions & Consumer Groups

The Kafka mental model. A topic is a named stream of messages. A topic is split into partitions for parallelism. A consumer group divides those partitions among its members. Three concepts, most of modern streaming.

Topics

A topic is just a named append-only log. Producers append messages; consumers read in order. Topics don't dictate who reads them — many consumers can read the same topic independently.

Partitions — the parallelism unit

A single log can only be appended sequentially. To scale, Kafka splits a topic into N partitions. Each partition is its own log. Messages are routed to a partition by key (same key always → same partition) or by round-robin (no key).

This is the same idea as database sharding — split data to parallelize.

Consumer groups — the work-sharing unit

Multiple consumers can join a group. Kafka assigns each partition to exactly one consumer in the group. If you have 4 partitions and 4 consumers, each gets one. 4 partitions and 2 consumers, each gets 2. 4 partitions and 8 consumers, 4 idle.

Different groups on the same topic each get the full stream — fan-out. Same group = load-balanced — fan-out one-of.

Try it

Tune number of partitions, consumers per group, and number of groups. See how messages distribute.

Key takeaways

  • Partition count = max parallelism. Adding consumers past partition count gets you idle consumers, not more throughput.
  • Partition by the right key. Same user's events on the same partition = guaranteed ordering for that user.
  • You can add consumers to a group at runtime; Kafka rebalances partition assignment.
  • Different consumer groups = independent consumption. Analytics + email service + audit log can all consume "orders" without affecting each other.
