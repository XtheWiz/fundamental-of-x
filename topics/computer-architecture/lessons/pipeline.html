Lesson 01

CPU Pipeline

A modern CPU isn't a single worker chewing through instructions one at a time. It's an assembly line — multiple instructions in flight, each at a different stage. When the line flows, you get one instruction per cycle. When it stalls, you get… less.

The five-stage classic

  1. Fetch — load the next instruction from memory.
  2. Decode — figure out what it is and what registers it reads/writes.
  3. Execute — run the ALU operation.
  4. Memory — load from / store to memory (if applicable).
  5. Writeback — write the result to a register.

In an idealized pipeline, instruction N is in stage 5 while N+1 is in stage 4, N+2 in stage 3, and so on. Five times the throughput of a non-pipelined design — same hardware, just kept busy.

The catch: hazards

When instruction N+1 needs a value that instruction N is still computing, the pipeline has to stall — insert bubble cycles where nothing useful happens. This is a data hazard. Real CPUs have forwarding (bypass paths) to hide many hazards, plus branch mispredictions (next lesson) that cause bigger stalls.

Key takeaways

  • A pipelined CPU completes ~1 instruction per cycle in the best case. Stalls reduce that.
  • Modern CPUs go further: superscalar (multiple instructions per stage), out-of-order (reorder around stalls), speculative (guess what's next, undo if wrong).
  • Bypass / forwarding hides most simple data hazards by sending a result directly to the next stage without waiting for writeback.
  • The performance enemy isn't computation — it's waiting (for data, for branches, for memory). The next 7 lessons are mostly about waiting.
