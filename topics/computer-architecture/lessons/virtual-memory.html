Lesson 05

Virtual Memory

Every process on your machine thinks it owns all of memory, starting at address 0. They can't all be right. Virtual memory is the elegant lie that makes them all seem right at once — and it's how process isolation works.

The trick

Each process has its own page table: a map from virtual page numbers to physical frame numbers. Every memory access goes through the MMU (Memory Management Unit), which translates virtual → physical on the fly.

Two processes can both have a pointer at virtual address 0x4000. The MMU's per-process page table sends them to different physical pages. They cannot accidentally see each other's memory.

The TLB

Walking the page table on every memory access would be too slow. The CPU has a tiny cache of recent translations called the TLB (Translation Lookaside Buffer). TLB hit: translation in 1 cycle. TLB miss: a full page-table walk, which is 50–200 cycles — and the table itself is in memory.

Try it

Click any virtual address. Watch the TLB get checked, then the page table walked. Repeat: notice the TLB hit cuts the work.

Key takeaways

  • Virtual memory enables isolation (processes can't read each other) and flexibility (the OS can swap, share, or grow memory transparently).
  • Page sizes are 4 KB by default. Bigger pages (2 MB, 1 GB) reduce TLB pressure for memory-heavy workloads (databases, ML).
  • Page faults — accessing memory that's not yet mapped to a frame — are how memory-mapped files and the OS's "RAM as cache for disk" work.
  • Spectre/Meltdown were attacks on the page-walk mechanism. Real security depends on this all being correct.
