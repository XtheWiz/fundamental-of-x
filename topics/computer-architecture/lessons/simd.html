SIMD & Vectorization

Single Instruction, Multiple Data. Your CPU has wide registers that can hold many numbers at once, and instructions that operate on all of them in a single cycle. It's an entire speedup hiding behind the right compiler flag.

The idea

A 32-bit add takes 1 cycle on a scalar ALU. A 256-bit SIMD add also takes 1 cycle — but it adds eight 32-bit values in parallel. Same hardware-clock, 8× the work done.

Common SIMD widths today:

  • SSE / AVX-128 — 128-bit, 4× floats per op
  • AVX2 / AVX-256 — 256-bit, 8× floats per op
  • AVX-512 — 512-bit, 16× floats per op
  • NEON (ARM) — 128-bit on phones & servers

Where you'll see it

Most compilers auto-vectorize simple loops at -O2 or -O3 — image filters, audio mixing, linear algebra, hashing, compression. Modern ML libraries (PyTorch's CPU paths, NumPy) live inside SIMD. Database engines use it for column-store aggregations.

Try it

Compare scalar vs SIMD multiplication of 8 numbers. Watch how 8 cycles collapse to 1.

Key takeaways

  • SIMD is essentially free CPU performance — if your loop fits the model (regular, branch-free, data-parallel).
  • Branches inside the loop kill auto-vectorization. Same with pointer chasing or unpredictable access patterns.
  • Modern languages help: numpy, Rust's std::simd, Go's compiler hints, Java's Vector API. You don't usually write assembly.
  • GPUs are SIMD on steroids — thousands of "lanes" running in lockstep. Same idea, vastly wider.
