Lesson 08

Performance Mental Model

Every previous lesson explained a single mechanism. Now compose them. Why is this loop slow? It's almost never "the algorithm" — it's a cache miss, a misprediction, a wasted cache line, or cores fighting over coherence.

The performance budget

For each clock cycle, the CPU is either:

  • Retiring instructions — actually doing work. Goal: maximize this.
  • Stalled on memory — waiting for cache miss to fill. The biggest source of "wasted" time.
  • Stalled on branch — pipeline flush after misprediction.
  • Stalled on coherence — multi-core invalidations.
  • Stalled on dependencies — data hazards the OoO engine can't hide.

Real CPU profiling tools (Linux perf, Intel VTune, macOS Instruments) sample these categories. Knowing which one dominates points you at the right fix.

Key takeaways

  • "It's slow" is never enough information. Why it's slow points to the fix.
  • For most CPU-bound code today, the binding constraint is memory bandwidth or cache misses — not arithmetic.
  • Mechanical sympathy: write code that lets the CPU do what it's good at — linear access, predictable branches, vectorizable inner loops.
  • And measure. Always measure. Intuition about performance is famously bad even for experts.
