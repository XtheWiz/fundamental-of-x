Branch Prediction

A pipelined CPU has to guess which way an if goes before it actually knows. Get it right, and the pipeline stays full. Get it wrong, and the speculatively- executed instructions get thrown out — that's a 15-cycle penalty on a modern CPU.

Why prediction is necessary

By the time the pipeline sees a branch in Execute, the instructions after it are already in Fetch and Decode. The CPU can't wait — it makes a guess (which direction is more likely?) and runs ahead. If the guess was right, no penalty. If wrong, flush everything in-flight and refill from the correct address. That's a big stall.

The predictor

Real CPUs use clever predictors, but the simple model is a two-bit saturating counter: it tracks recent outcomes for each branch, getting more "confident" the more consistent the history is. Repetitive branches (loop conditions) converge to near-100% accuracy; truly random ones stay around 50%.

Try it

Run a stream of branches. Predictor accuracy starts at 0% and converges. Toggle the pattern to see how predictability matters.

Key takeaways

  • Modern branch predictors hit ~95–99% accuracy on typical code. That's part of why CPUs are so fast.
  • The famous "sorted vs unsorted array" benchmark on Stack Overflow is a branch prediction story — sorted data makes the comparison predictable.
  • Mispredictions cost ~15–20 cycles each. A tight loop with unpredictable branches can be 5× slower than the same loop without branches.
  • Branchless code (using conditional moves, lookup tables, or arithmetic instead of if) sidesteps the predictor entirely. Useful in performance-critical inner loops.
