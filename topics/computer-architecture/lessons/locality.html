Memory Locality

The CPU doesn't read individual bytes — it reads 64-byte cache lines at a time. So the layout of your data on a single read decides whether the next 63 bytes are useful or wasted. Two programs with the same algorithm can differ 10× because of this.

The two layouts

  • Array of Structs (AoS) — the natural OOP layout. [{x, y, z, w}, {x, y, z, w}, …]. Each struct's fields are together.
  • Struct of Arrays (SoA) — split each field into its own array. {x: [...], y: [...], z: [...], w: [...]}. All x's are together.

Which is faster depends entirely on the access pattern:

  • "sum every x" — SoA wins. A single cache line holds 16 consecutive x values; AoS wastes 3/4 of every line on y, z, w.
  • "compute on one whole struct" — AoS wins. One cache line gets you the whole struct.

Try it

Choose a query, choose a layout, watch how many cache lines actually had to be loaded.

Key takeaways

  • Spatial locality — bytes near each other on disk/in memory get loaded together. Sequential access is fast; random access is slow.
  • Temporal locality — bytes accessed recently are likely to be accessed again soon. Caches exploit both kinds.
  • Choosing AoS vs SoA is a real design decision in game engines, databases, and data-processing systems. Many DBs (ClickHouse, DuckDB) are SoA at heart — that's also why column stores exist (recall Database lesson 3).
  • The same trade-off shows up at every level: cache lines, disk blocks, network packets, file formats. Anywhere you fetch in chunks, alignment to access pattern matters.
