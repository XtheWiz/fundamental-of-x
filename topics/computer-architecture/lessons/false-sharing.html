Multi-core & False Sharing

Four cores, four counters, no shared writes — should scale perfectly, right? It can be 10× slower than one core. Welcome to false sharing: the most expensive 64 bytes of memory in concurrent code.

The coherence protocol

Each core has its own L1 cache. When two cores need the same cache line, they negotiate: MESI (Modified / Exclusive / Shared / Invalid) tracks who has what.

  • Modified: this core has the only copy, and it's been written.
  • Exclusive: this core has the only copy, clean.
  • Shared: multiple cores have read-only copies.
  • Invalid: this core's copy is stale and must be re-fetched.

Writes are the expensive ones: when a core writes, all other cores' copies of that line are invalidated. They have to re-fetch (often from RAM, dozens of nanoseconds).

False sharing

If four threads each increment their own counter, but all four counters happen to sit on the same 64-byte cache line, every write invalidates the other three cores' copies. The line ping-pongs between cores; cycles burn doing nothing.

The fix: pad each counter to its own cache line. Same logic, 10× faster.

Try it

Run 100 increments per thread. Compare shared layout vs padded layout. Watch the cache-line state changes.

Key takeaways

  • "Per-thread state" only scales if each thread's state lives on its own cache line.
  • Common patterns that hit this: per-thread counters, per-thread random state, hot atomics in metrics.
  • The pad-to-cache-line trick is standard in high-performance libraries (Java's @Contended, C++'s alignas(64), Rust's repr(align(64))).
  • You can't see false sharing in normal profiling — CPU samples look like idle waits. Look for it whenever multi-core scaling is mysteriously sublinear.
