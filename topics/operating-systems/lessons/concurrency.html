Lesson 06

Concurrency Primitives

Two threads, one variable. They take turns incrementing it. Sometimes they collide and the variable loses a write. This is the bug at the heart of every "we have a race condition" postmortem.

The race

The instruction counter++ is actually three steps:

  1. Load counter into a register
  2. Add 1
  3. Store back to memory

If thread A finishes step 1 (sees 5), then thread B does all three (sets it to 6), then A continues with step 3 (writes 6 back) — one increment was lost. The classic read-modify-write race.

The primitives

  • Mutex (mutual exclusion lock) — only one thread inside the critical section at a time. The basic tool.
  • Semaphore — count of available "permits"; threads acquire/release. Generalizes mutex (semaphore with capacity=1).
  • Condition variable — wait until some condition becomes true, signalled by another thread. Pairs with a mutex.
  • Atomic operation — single CPU instruction does load+modify+store as one indivisible step. Faster than a mutex for simple cases.

The next bug: deadlock

Now you have two mutexes. Thread A locks M1, then M2. Thread B locks M2, then M1. If both grab their first lock at the same time, they each wait forever for the other's. Welcome to deadlock.

Standard avoidance: always acquire locks in a consistent global order. Easy to say, hard to enforce in a large codebase.

Try it

Run two threads racing on a counter. Toggle a mutex. Then try the two-mutex deadlock scenario.

Key takeaways

  • Race conditions are silent — your code "mostly works" until it doesn't. Reproducing them is famously hard.
  • Mutexes are the right default. Lock-free / atomic-based code is faster but much harder to write correctly.
  • Deadlock has four required conditions (Coffman): mutual exclusion, hold-and-wait, no preemption, circular wait. Break any one and deadlock is impossible.
  • Modern languages offer higher-level escape hatches: channels (Go, Rust), actors (Erlang), structured concurrency (Swift, Python's asyncio), CSP-style coordination.
