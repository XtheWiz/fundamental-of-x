The Scheduler

Hundreds of processes, eight CPU cores. Someone has to decide which process runs next, on which core, for how long. That "someone" is the scheduler — arguably the most important piece of code in the kernel.

The classic policies

  • FCFS (First Come, First Served) — simple, deterministic, terrible if a long job arrives first.
  • Round Robin — give every process a fixed time slice, rotate. Fair, but adds context-switch overhead.
  • SJF (Shortest Job First) — pick the shortest remaining work. Minimizes average wait time. Requires knowing the future (so in practice we estimate).
  • Priority — higher priority always runs first. Risk: starvation of low-priority work.

What Linux actually uses

The Completely Fair Scheduler (CFS, default since 2007) doesn't use any of those directly. It tracks virtual runtime per task — basically "how much CPU time you've gotten, adjusted for priority" — and always picks the task with the least vruntime. The result is approximately fair sharing, with priorities as a weighting.

The 2024-onwards default is EEVDF (Earliest Eligible Virtual Deadline First), which is conceptually similar but adds latency bounds.

Try it

Four jobs of different lengths. Pick a policy and watch the CPU schedule them. Compare average wait and turnaround time.

Key takeaways

  • No scheduler is best for every workload. Real systems balance throughput, latency, fairness, and priority.
  • Context switches aren't free — a few microseconds each. Tiny time slices = high overhead.
  • Interactive tasks (your editor) deserve low latency. Batch jobs (compilation) tolerate it. CFS infers which is which from sleep patterns.
  • Real-time schedulers exist for tasks with hard deadlines (audio, robotics). They prioritize predictability over throughput.
