Inter-Process Communication

Processes are isolated by design — but real applications need them to coordinate. Unix offers half a dozen ways for processes to talk, each with very different speed, scope, and ergonomics.

The menu

  • Pipe / named pipe (FIFO) — byte stream, parent↔child or via filesystem. Simple, slow-ish (kernel buffer).
  • Unix domain socket — like a TCP socket but local. Datagram or stream. Used by Docker, X11, dbus, container runtimes.
  • Shared memory — a region of memory mapped into both processes' address spaces. Fastest possible IPC; needs your own synchronization.
  • Message queue (POSIX) — kernel-managed mailbox. Discrete messages, priorities. Less common now.
  • Signals — tiny async notifications (one of ~30 numbered signals). For control, not data.
  • TCP loopback — same protocol as remote networking, just on 127.0.0.1. Works everywhere; slowest local option.

Try it

Pick a mechanism, see relative throughput, when to reach for it, and what a small message exchange looks like.

Key takeaways

  • Shared memory is the fastest but the most dangerous — no kernel mediation means you must synchronize yourself (last lesson's mutexes).
  • Unix domain sockets are the modern default for "two processes on the same box need to talk." Docker, gRPC, browsers all use them.
  • Signals are not a data channel — they're an interrupt. Use them to say "shut down" or "reload config," not "here's a request."
  • If your two "processes" are actually two threads, just use shared memory (your variables) plus a mutex. The IPC menu is for when you can't.
