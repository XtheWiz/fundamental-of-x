Processes vs Threads

Both are units of execution. The difference is what they share. Threads share an address space; processes don't. That single distinction explains why threads are cheap to create and dangerous to reason about — and why processes are the opposite.

What a process owns

A Linux process owns:

  • An address space (its own virtual memory layout).
  • One or more threads (with shared stacks-in-the-same-address-space).
  • A file descriptor table (open files, sockets, pipes).
  • A process ID (PID), parent PID, credentials, signal handlers.
  • Working directory, environment variables, controlling terminal.

Threads within a process share all of those except their own stack and a few thread-local bits. That's the speed and the risk in one sentence.

fork(), exec(), and the family tree

Unix has a beautifully minimal API for creating processes:

  • fork(): clone this process. Returns 0 in the child, the child's PID in the parent. Two processes, identical state, single line of code.
  • exec(): replace this process's program with a different one. Same PID, same FDs (usually), entirely different code.

"fork then exec" is how every shell launches every command. Modern Linux also has clone(), which lets you control exactly what to share (the basis of threads + containers).

Key takeaways

  • Processes are isolated; threads are shared. Pick threads when you need to share data fast; pick processes when you can't afford a crash to take down everything.
  • Creating a thread is ~10× cheaper than a process. fork() is fast because Linux uses copy-on-write — children only diverge from parents lazily.
  • Threads can corrupt each other's data via races (next lessons cover concurrency). Processes can't — at most they can leak FDs or kill each other with signals.
  • "Process" and "thread" are kernel concepts. Languages add their own: green threads, fibers, goroutines, coroutines — all built on top.
