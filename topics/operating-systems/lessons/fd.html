File Descriptors & Pipes

"In Unix, everything is a file" is half true. The full truth is "in Unix, everything is a file descriptor" — a small integer indexing into a per-process table of open streams. Files, sockets, pipes, terminals — same API.

The FD table

Each process has a table. By convention:

  • fd 0 = stdin
  • fd 1 = stdout
  • fd 2 = stderr
  • fd 3, 4, 5… = whatever you open next

open() returns the lowest-numbered free FD. close() frees that slot. dup2(old, new) makes new point to the same underlying stream as old — the trick that makes shell redirection work.

Pipes are FDs too

pipe() creates two new FDs: a read end and a write end. They share a kernel buffer. Two processes (parent + forked child, typically) can communicate by writing to one end and reading from the other.

cat foo.txt | grep bar is exactly this: the shell forks cat, redirects its stdout (fd 1) to the pipe's write end via dup2, forks grep, redirects its stdin (fd 0) to the pipe's read end. Both programs are oblivious — they think they're using stdin and stdout normally.

Try it

Open files. Pipe two processes. Redirect stdin/stdout. Watch the FD table rewire.

Key takeaways

  • FDs are per-process. fd 3 in one process and fd 3 in another are unrelated.
  • FDs are inherited across fork() — that's how parent-child IPC works.
  • The same read/write API works for files, pipes, sockets, devices. That uniformity is one of Unix's biggest design wins.
  • Modern alternatives (Windows handles, io_uring fds) extend the idea but the basic model has survived 50 years for a reason.
