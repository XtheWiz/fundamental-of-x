Syscalls

Your program lives in user mode; the kernel lives in kernel mode. Crossing that boundary — to write a byte, open a file, send a packet — happens via a syscall. It's the bedrock of every API your language gives you.

Two worlds, one trap door

The CPU has privilege levels. Ring 3 (user mode) can't touch hardware, can't see other processes' memory, can't change page tables. Only ring 0 (kernel mode) can. The user/kernel divide is the OS's main security boundary.

To cross over, user code executes a special instruction — syscall on x86-64, svc on ARM. The CPU saves the user state, jumps to a kernel entry point, runs the handler, then jumps back. It looks like a function call; it's actually a context-switch and a privilege jump.

The syscall table

Each syscall has a number. read is 0, write is 1, open is 2, fork is 57. Your library (libc) puts the number in a register, the arguments in other registers, and traps. The kernel uses the number to index into a table of function pointers.

Try it

Pick a syscall. Step through the transition: user code → libc wrapper → trap → kernel handler → return.

Key takeaways

  • A syscall is expensive — hundreds of cycles for the privilege change alone. Buffering (e.g., stdio) and batching (e.g., writev, io_uring) exist to amortize the cost.
  • Modern fast paths bypass the trap: vDSO exposes a few "cheap" calls (gettimeofday) as user-space code; io_uring uses a shared-memory ring buffer so a thousand I/Os can be issued without traps.
  • Run strace your-program in a terminal to watch every syscall it makes. Eye-opening — most "innocent" programs make thousands per second.
  • The syscall API is the kernel's most stable promise. Linux has added syscalls over decades but never breaks old ones — that's why binaries from 1995 still run.
