Lesson 08

Containers (namespaces + cgroups)

Containers are not virtual machines — they're regular Linux processes wearing a costume. Two kernel features (namespaces and cgroups) do the entire trick. Once you see them, Docker stops being magic.

Namespaces — "what can this process see?"

A namespace is a kind of tunnel-vision. The kernel maintains multiple instances of certain global resources; each process sees only its own slice. Linux ships six:

  • PID — process IDs. Inside, your container's main process is PID 1.
  • NET — network interfaces, IP addresses, routing tables.
  • MNT — mount points; your filesystem view.
  • UTS — hostname and domain name.
  • IPC — System V message queues and shared memory segments.
  • USER — UIDs and GIDs (uid 0 inside can map to an unprivileged uid outside).

cgroups — "what can this process use?"

Control groups limit and account for resources. CPU shares, memory ceiling, block-I/O bandwidth, # of PIDs. They're how Docker's --memory 512m and Kubernetes resource limits actually work.

Try it

Start a process and toggle namespaces on. Watch what the process can still see of the host — and what disappears.

Key takeaways

  • A "container" is just a normal Linux process running with namespaces + cgroups applied. Run ps aux on the host and you'll see them — same PID space (from the host's view), with a confined view of the world.
  • Containers share the host kernel. A kernel exploit inside one container can theoretically affect the host. That's the main difference from VMs (separate kernels).
  • Docker, podman, containerd, and Kubernetes are all orchestrators on top of the same kernel primitives. You can do everything they do with unshare + chroot + cgroup files (if you have the patience).
  • The user namespace is the modern security win: it lets containers run "as root" while having no actual host privileges.
