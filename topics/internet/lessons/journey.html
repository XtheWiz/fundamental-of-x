Lesson 01

Journey of a Request

You type a URL and hit enter. Less than a second later a webpage appears. In between is one of the most over-engineered, layered, interdependent assembly lines humans have ever built — and it almost always works.

The big picture

Five phases. Each one is its own lesson in this series, but here's the assembly:

  1. DNS — turn the domain name into an IP address (1–100ms, depending on cache).
  2. TCP — three-way handshake to open a connection (1 RTT).
  3. TLS — secure handshake to set up encryption (1 more RTT in TLS 1.3).
  4. HTTP — actually send the request, wait for the server to compute and return a response.
  5. Render — browser parses HTML/CSS/JS and paints pixels.

For repeat visits, DNS is cached and the connection may be reused (or 0-RTT'd with TLS resumption). That's why your second visit is so much faster.

Try it

Pick a scenario and step through. Watch a waterfall chart fill in as each phase completes.

Key takeaways

  • Every phase is at least one round trip. Geography (RTT) is the dominant cost on a cold visit.
  • Caches everywhere: DNS, TCP connections, TLS sessions, HTTP responses, browser memory. Each cache layer skips a phase.
  • "Performance" is mostly about making the first visit fast and the second visit nearly free.
  • The dig into individual phases happens in the next 7 lessons.
