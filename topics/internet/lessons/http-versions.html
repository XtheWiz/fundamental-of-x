Lesson 07

HTTP/1.1, HTTP/2, HTTP/3

Same protocol, three radically different implementations. Each new version was driven by one specific bottleneck — and modern page loads are a hundred times faster because of them.

The bottleneck progression

  • HTTP/1.1 (1997) — one request at a time per connection. Browsers worked around this by opening up to 6 parallel TCP connections per host, but it's still a lot of round trips.
  • HTTP/2 (2015) — multiplexed streams over one TCP connection. Many requests in flight at once. Big win for page loads with many small assets. BUT one lost TCP packet stalls every stream — "TCP head-of-line blocking."
  • HTTP/3 (2022) — same multiplexing idea, but over QUIC (built on UDP). Each stream has its own reliability. Lost packets only stall their own stream. Plus 0-RTT for repeat connections.

Try it

Race the three protocols loading 10 resources. Crank packet loss and watch HTTP/2 collapse while HTTP/3 keeps going.

Key takeaways

  • HTTP/2 fixed the "many connections" problem. HTTP/3 fixed the "one lost packet stalls everything" problem.
  • HTTP/3 is over UDP not because UDP is faster — but because TCP is fundamentally serial. QUIC builds independent streams on top of UDP.
  • Modern CDNs (Cloudflare, Fastly) speak HTTP/3 already. Most browsers negotiate it automatically when available.
  • Servers still need HTTPS to use HTTP/2 or HTTP/3 — no clear-text version was ever standardized.
