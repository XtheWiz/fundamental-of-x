TCP vs UDP

Two transport protocols, one network. TCP cares about every byte arriving in order; UDP doesn't care if some go missing. The tradeoff is reliability vs latency, and the right choice depends entirely on what you're sending.

The two philosophies

  • TCP — connection-oriented. Three-way handshake to start. Every segment is numbered, ACKed, and retransmitted if lost. Bytes always arrive in order. Pays latency for reliability.
  • UDP — connectionless. Send a datagram, hope for the best. No ordering, no retry, no acks. If a packet drops, your app deals with it (or ignores it). Lowest possible overhead.

Rule of thumb: anything that breaks if you lose a byte → TCP. Anything that breaks if you wait → UDP.

Try it

Send 10 packets over both. Crank the packet loss slider and watch how each protocol responds.

Key takeaways

  • TCP guarantees delivery and order at the cost of higher latency (waiting for retransmits) and per-connection state.
  • UDP is "best effort." Lost packets are lost — but the protocol gets out of your way.
  • Use TCP for: web (HTTP/1, /2), file transfer, email, SSH — anything that requires every byte.
  • Use UDP for: DNS queries, real-time games, video calls (WebRTC), DNS, NTP — anything where late data is worse than missing data.
  • QUIC (HTTP/3) is built on UDP but adds its own reliability — best of both worlds, at the cost of complexity.
