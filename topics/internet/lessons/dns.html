DNS — Names to Numbers

Computers talk in IP addresses (185.199.108.153). Humans talk in names (fundamentalofx.com). DNS — the Domain Name System — is the worldwide phone book that translates one into the other, in milliseconds, billions of times a second.

The hierarchy

DNS is a tree. The root is at the top, then top-level domains (TLDs) like .com, then second-level domains like github, then subdomains. To resolve a name your machine asks a recursive resolver (usually your ISP, or 1.1.1.1 / 8.8.8.8), which walks the tree on your behalf.

The walk has four hops in the worst case:

  1. Recursive resolver — does the work for you. Caches aggressively.
  2. Root nameserver — "who handles .com?" Returns TLD servers.
  3. TLD nameserver — "who handles github.com?" Returns authoritative servers.
  4. Authoritative nameserver — "what's github.com's IP?" Returns the answer.

In practice the resolver caches most of these answers, so a typical lookup is just one round trip to the resolver.

Try it

Pick a domain and resolve it. Toggle cache to see how shortcuts kick in.

Key takeaways

  • DNS is hierarchical: root → TLD → authoritative. The recursive resolver does the walk for you.
  • Caching is everything. A hot domain is one ~1ms lookup; a cold one can take 30–100ms across four hops.
  • TTL (Time To Live) on each record controls how long resolvers cache. Low TTL = faster propagation, more DNS traffic.
  • DNS over UDP (port 53) by default, falls back to TCP for big answers, now also DoH (DNS-over-HTTPS) for privacy.
