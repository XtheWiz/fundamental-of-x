TCP Handshake & Flow

TCP turns IP's "best effort" into something you can actually build on: every byte numbered, every segment ACKed, lost ones retransmitted. The price is one round trip of handshake before you can send anything.

The three-way handshake

  1. Client → Server: SYN seq=x "let's talk, my starting number is x."
  2. Server → Client: SYN-ACK seq=y ack=x+1 "got it, my starting number is y."
  3. Client → Server: ACK ack=y+1 "we're connected."

Now both sides agree on starting sequence numbers, and data can flow. Every byte sent counts up, every byte received is ACKed.

Sliding window & retransmission

TCP doesn't send one packet and wait for an ACK before sending the next — that'd be slow. It sends a window of unacked segments, then slides forward as ACKs arrive. If an ACK doesn't show up in time (the retransmission timeout), the sender assumes loss and resends from the gap.

Key takeaways

  • The handshake is one full RTT before any data can flow. That's why TCP feels slow over high-latency links.
  • The sliding window lets TCP keep multiple segments in flight, dramatically improving throughput.
  • Lost ACKs cause retransmissions, which cause TCP to back off (slow start, congestion avoidance) — that's why one bad packet can stall a whole flow.
  • TCP teardown is similar: FIN / FIN-ACK / ACK to gracefully close.
