TLS Handshake

The lock icon in your browser hides a small miracle. Two strangers — your laptop and a server you've never spoken to — agree on a shared secret, in public, over an internet packed with eavesdroppers. TLS 1.3 does this in one round trip.

The problem TLS solves

Three guarantees, all at once:

  • Confidentiality — eavesdroppers can't read the contents
  • Integrity — they can't modify packets without being detected
  • Authentication — you're really talking to fundamentalofx.com, not an impersonator

The clever bit: TLS achieves all three using public-key crypto (slow, but works without prior secrets) to set up symmetric crypto (fast). The handshake is the negotiation.

Try it

Step through the TLS 1.3 handshake one message at a time. Each step shows what's being sent and what each side now knows.

Key takeaways

  • TLS 1.3 is 1-RTT: the client sends a key share in ClientHello, so the server can derive the shared secret on its first reply.
  • The certificate (signed by a CA your machine trusts) is how the server proves identity. Without that, you'd have no idea who you're encrypting to.
  • After the handshake, all data uses fast symmetric encryption (AES-GCM, ChaCha20-Poly1305). The slow asymmetric crypto is only used at the start.
  • Forward secrecy: each session uses ephemeral keys, so even if the server's long-term key leaks tomorrow, today's traffic is still safe.
