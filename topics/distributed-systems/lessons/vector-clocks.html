Vector Clocks

There is no global clock in a distributed system — clocks drift, NTP only gets you within milliseconds, and "the same time" is meaningless across continents. Vector clocks sidestep wall time entirely and track causality: who saw what, when.

The rules

Each node N keeps a vector of integers — one slot per node:

  1. Local event on N: increment V[N].
  2. Send a message from N: increment V[N], attach the whole vector to the message.
  3. Receive a message on M: V[M]++, then merge: V = max(V_local, V_received) element-wise.

Now you can compare two events' vectors:

  • A < B (A happened-before B): every entry of A ≤ B, and at least one is strictly less.
  • A == B: identical (same event).
  • Concurrent: neither A < B nor B < A — they happened independently.

Try it

Three nodes. Trigger events on each, send messages between them, watch the vectors evolve. Compare any two events to see causality.

Key takeaways

  • Vector clocks make "happened-before" decidable without wall clocks. They're the canonical tool for distributed causality.
  • Cost is O(N) bookkeeping per message, where N is the number of nodes. Fine for small clusters; awkward for huge ones (replaced by hybrid logical clocks or dotted version vectors at scale).
  • Concurrent events are the interesting case — they're where you need conflict resolution (last-writer-wins, CRDTs, application-level merge).
  • Used in DynamoDB, Riak, version control systems (Git's DAG is essentially vector-clock-like), and CRDTs.
