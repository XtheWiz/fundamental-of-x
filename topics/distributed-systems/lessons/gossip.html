Gossip & Failure Detection

How does a 1000-node cluster keep everyone informed about who's alive? Not with a central coordinator (too brittle), not with broadcast (too noisy) — but with gossip. Each node whispers to a few random neighbors. The news spreads exponentially.

The protocol

Every gossip interval (say, 1 second), each node:

  1. Picks k random peers (typically 1-3).
  2. Sends them its current state — membership list, version vectors, health status.
  3. Each receiver merges the gossip into its own state.

That's it. No coordinator. No global view. Yet:

  • Convergence is exponential. After roughly logk+1(N) rounds, everyone knows. For 1000 nodes with k=3, that's ~5 rounds.
  • It's robust. Lose any subset of messages; the rest still converge.
  • Failure detection rides along. If a node hasn't shown up in anyone's recent gossip, peers suspect it. Eventually they declare it down.

Try it

Drop a rumour on one node, then watch it spread by gossip. Tune the fan-out and see convergence time change.

Key takeaways

  • Gossip is the membership / failure-detection foundation of Cassandra, Riak, Consul, Serf, Akka Cluster, and the BGP routing tables that run the internet.
  • The math is robust: even with 50% message loss, gossip still converges (just slower).
  • Phi-accrual is the failure detector most large systems use — it doesn't declare a node "dead" or "alive" but reports a continuous suspicion level.
  • Trade-off: gossip is eventually consistent on membership info. You won't know about a new node for a few seconds after it joins.
